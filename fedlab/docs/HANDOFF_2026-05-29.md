## Updated idea outline

FedLab starts as a **Bags-powered verified model registry**. It begins with a
curated array of models, each verified by a benchmark *validation packet* — not a
complex decentralized training network.

1. A registry tracks models, each with a domain, benchmark, score, and status.
2. Each domain has a validation packet: dataset summary, metric, evaluation
   script, current best score, leakage checks.
3. A passing validation produces a validation hash → a public proof of validation
   that can be anchored on Solana.
4. Bags supplies the project/token traction layer on top of the registry.

Initial tracks: **SWE-bench++** (software engineering), **TabPFN** (tabular),
**Reservoir Finance** (financial forecasting).

## Backend / API plan and endpoints

FastAPI app at `aggregator/server.py`, file-backed state. New endpoints:

- `GET  /registry/models` — list models
- `GET  /registry/models/{model_id}` — one model
- `POST /registry/submit` — queue a model; returns a `candidate` row (no
  validation run yet)
- `GET  /validation-packets` — list packet metadata
- `GET  /validation-packets/{packet_id}` — one packet
- `GET  /integrations/status` — Bags + Solana config status
- `GET  /integrations/bags/project` — project stats (mock if unconfigured)
- `GET  /integrations/bags/token` — token stats (mock if unconfigured)

Existing: `/health`, `/model/latest`, `/leaderboard`, `/contribute`.

## Bags API integration plan + env vars

`integrations/bags.py`: `get_project_stats()`, `get_token_stats()`,
`get_integration_status()`. Returns `configured: false` / `mode: "mock"` when env
is missing. Endpoint paths are configurable placeholders; no private API is called.

Env: `BAGS_API_BASE_URL`, `BAGS_API_KEY`, `FEDLAB_BAGS_PROJECT_ID`,
`FZIQ_TOKEN_ADDRESS`.

## Web3 / Solana integration plan + env vars

`integrations/solana.py`: `is_valid_wallet()`, `build_validation_proof(model)`
(sha256 over validation event → anchorable hash), `check_onchain_proof(hash)`,
`get_integration_status()`. **No contract, no mint, no transfer.**

Env: `SOLANA_RPC_URL`, `FZIQ_TOKEN_ADDRESS`.

## Caveats / limitations

- Seed scores are illustrative targets, labeled demo/candidate. Do not present
  them as measured.
- SWE-bench++ is a packet *definition*; the execution harness is not built.
- Integrations return mock/not-configured data until credentials are supplied.
- No Solana contract or token mechanics are implemented (intentional).
- State is file-backed and single-process — fine for the hackathon, not for scale.

## Autonomous Agent Operating Model

Added an **advisory** autonomous-agent operating model to drive the launch with
local Ollama. Agents produce plans, drafts, copy, research, and findings; the human
owner implements anything sensitive. See `../agents/README.md` and
`GITHUB_OPERATING_MODEL.md`.

### Ollama local setup (outline)

```bash
ollama serve                     # daemon (usually already running)
ollama pull llama3.1:8b          # light; or qwen2.5:14b / deepseek-r1:14b
ollama run llama3.1:8b "hi"      # confirm it responds
```

The runner talks to `http://localhost:11434` (override with `OLLAMA_HOST`).

### Roles (in `agents/prompts/`)

- **CTO Architect** — scope control, decision log, roadmap, ruthless cuts.
- **Security Reviewer** — threat model, secrets, auth boundaries, wallet-proof and
  financial-language review. Findings only; never implements auth/crypto.
- **Backend API** — endpoint + OpenAPI design, registry persistence plan. No admin
  auth or production wallet verification without owner review.
- **Validation Engineer** — packet design, leakage checks, reproducibility. Never
  runs untrusted code or loads untrusted models.
- **Frontend Product** — UI copy, submit-flow wireframes, honest status labels.
- **Growth Operator** — launch plan, Discord/X/GitHub outreach, interview scripts.
- **Bags/Web3 Researcher** — maps Bags API/Solana proof approach + env vars. No
  contracts, mints, or transactions.
- **Fundraising / Applications** — Bags application answers, traction narrative,
  demo script.

Each prompt specifies Mission, allowed Inputs, expected Outputs, Forbidden actions,
Escalation conditions, and a GitHub interaction model (propose issues/PR plans, not
sensitive code).

### Running an agent (advisory only)

`agents/run_agent.py` reads a role prompt + your task, calls local Ollama, and
writes the result to `agents/out/`. It does **not** modify the repo, run shell
commands, touch git/GitHub, or handle secrets.

```bash
cd fedlab
python agents/run_agent.py --role cto_architect --task "Cut scope for the Bags demo."
python agents/run_agent.py --role security_reviewer --task-file notes.md \
  --context-file docs/HANDOFF_2026-05-29.md --model qwen2.5:14b
python agents/run_agent.py --role backend_api --task "..." --dry-run   # inspect prompt
```

### Owner-only sensitive tasks (NOT done by agents)
